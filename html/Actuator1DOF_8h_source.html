196  /*
197  There are subclasses of RotationalActuator that want to create their own
198  versions of the input shaft and the internal connector. For example, the
199  hydraulics package contains the ImpellerActuator, which uses a FlowUnit
200  for input.
201 
202  Any subclass that uses the dont_create_input tag when constructing the
203  RotationalActuator base class must implement get- and setEnable.
204 
205  This part of the inheritence tree could use some redesign. There should
206  be one base class, inheriting from Actuator1DOF, that knows about hinges
207  and how to get Jacobians from them, and a collection of subclasses
208  derived of off that that knows about the type of the input and the custom
209  constraint that binds the input PhysicalDimension to the
210  ActuatorBodyUnits.
211 
212  Perhaps we should even move the concept of input unit and internal
213  constraint up a bit more and put it inside Actuator1DOF. In that case we
214  don't have to deal with the oddities of the current setEnable
215  implementation, which is just bad. A base class implementation does the
216  wrong thing for the derived classes.
217 
218  Not sure if the name RotationalActuator should be used for the base class
219  or the subclass that has a RotationalUnit input. The latter is more
220  consistent with the current implementation and a better user-level name
221  (the base class is not user-level), but what should the base class be
222  named?
223  */

225  static dont_create_input_t dont_create_input;
